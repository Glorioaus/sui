## 退款对冲（`reconcile_refunds`）

跨文件、跨账户匹配消费与退款，匹配成功双方删除。

**两轮匹配**：

### 第一轮：精确匹配
- 条件：**同商户（标准化后） + 同金额**（容差 ±0.01 元）
- 商户标准化：移除前缀（消费/支付/转账/退款）+ 去空白连字符 + 转小写

### 第二轮：模糊匹配
- 仅对**脱敏商户名**（含 `**`，如 `天猫**营`）或**人名**（2-4 个汉字）触发
- 条件：**同金额 + 日期接近**（±30 天，因为退款可能很晚）

**退款识别**：收入侧交易，其 description/category/subcategory 任一含「退款」即视为退款。

**输出**：精确匹配对数 + 模糊匹配对数 + 删除记录数。

## 转账识别（`identify_transfers`）

识别储蓄卡→信用卡/支付宝/微信的转账，把支出转为转账记录。

### 触发条件

储蓄卡支出 + 描述含信用卡关键词，**且未被分类为「金融保险-按揭还款」**。

### 关键词映射（`TRANSFER_KEYWORDS`）

| 描述含 | 转账目标 |
|--------|---------|
| `中信` | 中信信用卡 |
| `招商` | 招商信用卡 |
| `浦发` | 浦发信用卡 |
| `建行信用` / `建设银行信用` / `建行卡` | 建行信用卡 |
| `花呗` | 花呗 |
| `京东白条` | 京东白条 |
| `微信` / `零钱` | 微信（⚠️ 须含转账语义标记，见下）|
| `支付宝` / `余额宝` | 支付宝（⚠️ 须含转账语义标记，见下）|
| `信用卡还款` / `还款` / `跨行还款` | None（需金额匹配确定目标） |

> ⚠️ **钱包类（微信/支付宝/零钱/余额宝）须同时含转账语义标记**（`充值`/`转入`/`零钱`/`余额宝`/`还款`/`提现`）才算转账；否则视为**经钱包的商户消费**（如「财付通-微信支付-美团」「支付宝-XX商户消费」）保留在支出，不误判为转账。信用卡/信贷类关键词不受此限制。

### 储蓄卡白名单（`DEBIT_ACCOUNTS`）

`农业银行`、`宁波银行`、`建行储蓄卡`、`工商储蓄卡`、`招商储蓄卡`、`农村信用合作社`

✅ 已修正：`ccb_parser.py` 的 account 名已从 `"建设银行信用卡"` 改为 `建行储蓄卡`（在此列表中），建行储蓄卡的还款支出可被正常识别为转账。

### 信用卡/钱包白名单（`CREDIT_ACCOUNTS`）

`中信信用卡`、`浦发信用卡`、`招商信用卡`、`建行信用卡`、`信用卡`、`花呗`、`京东白条`、`微信`、`支付宝`

### 两轮匹配

1. **第一轮（有明确目标）**：描述含具体银行关键词（如"中信"）→ 直接标记转账到对应信用卡
   - 尝试匹配信用卡的 `__REPAYMENT__` 收入记录（金额+±3天），匹配则删除两端
   - 未匹配但目标明确，仍标记为转账

2. **第二轮（需要匹配）**：描述含"跨行还款"/"还款"但无具体银行名
   - 通过金额+日期（±3天）匹配信用卡的 `__REPAYMENT__` 收入记录
   - 匹配成功：识别为转账到对应信用卡
   - **匹配失败：标记为转账到通用"信用卡"**

### 子分类规则

| 转账目标 | 子分类 |
|---------|--------|
| 支付宝 / 微信 | 充值 |
| 信用卡 / 花呗 / 京东白条等 | 还款 |

## 亲属卡处理（`process_family_card`）

处理微信"亲属卡交易"和支付宝"亲友代付"，把对应的银行卡支出重分类为「其他杂项-{使用者}支出」。

### 标记来源

- 微信：`category="__FAMILY_CARD__"` + `transaction_type="__MARKER__"`，subcategory 存使用者名
- 支付宝：同上机制，"交易对方"即使用者姓名（如"张颖"）

### 处理逻辑

1. 收集所有标记交易（`__FAMILY_CARD__` 或 `__MARKER__`）
2. 统计各银行账户是否有数据
3. 对每个标记，在银行卡交易中查找**日期 + 金额完全匹配**的交易（若标记指定具体银行，账户也要匹配）
4. 三种结果：
   - **匹配成功**：重分类银行卡交易为「其他杂项-{使用者}支出」，删除标记
   - **未匹配 + 银行有数据**：删除标记（避免与银行账单重复）
   - **未匹配 + 银行无数据**：保留标记作为「其他杂项-{使用者}支出」（标记本身转为正式交易）

### 作用

区分本人消费和亲属消费，避免重复记账。

## 处理顺序

`merge.py` 的 `merge_excel_files` 严格按序执行：

1. `reconcile_refunds` —— 退款对冲
2. `identify_transfers` —— 转账识别
3. `process_family_card` —— 亲属卡处理
4. `sort_transactions` —— 按日期排序
5. `ExcelGenerator` 生成最终 Excel

**顺序敏感**：转账识别依赖退款对冲先清理（避免对冲掉的交易再被识别为转账）；亲属卡处理依赖前两步完成。

## LLM 何时介入

以下场景留作未来 LLM 介入点（主流程当前不调用，仅走确定性脚本）：

1. **转账目标为通用"信用卡"**：`identify_transfers` 第二轮匹配失败时，可综合 description+金额+日期+账户做归属判断
2. **亲属卡未匹配**：标记未在银行卡找到精确匹配，但金额/日期接近时，可判断是否为同一笔

**始终不交给 LLM**：金额精确匹配、日期范围计算、按揭还款排除——这些是纯确定性逻辑。
